# 程式設計與軟體工程：簡要規則集

軟體工程是應用系統化、規範化、可量化的方法來開發、運作和維護軟體。本規則集涵蓋了編寫高品質程式碼與構建可靠系統的核心原則。

優秀的軟體不僅是可以運行的代碼，更是易於閱讀、維護和擴展的資產。

## 核心原則

1. **SOLID 原則**：物件導向設計的五大原則，有助於建立易於維護和擴展的系統。
   - **S**RP (單一職責原則)：一個類別應該只有一個改變的理由。
   - **O**CP (開放封閉原則)：軟體實體應該對擴展開放，對修改封閉。
   - **L**SP (里氏替換原則)：子類別應該能夠替換其父類別而不影響程式的正確性。
   - **I**SP (介面隔離原則)：不應強迫客戶端依賴它們不使用的方法。
   - **D**IP (依賴反轉原則)：依賴於抽象，而不是具體實作。
2. **KISS 原則 (Keep It Simple, Stupid)**：保持系統簡單。複雜度是軟體開發的敵人，簡單的設計更易於理解和維護。
3. **DRY 原則 (Don't Repeat Yourself)**：避免邏輯重複。將重複的邏輯提取為函數或模組，以減少維護成本和錯誤風險。
4. **YAGNI (You Aren't Gonna Need It)**：不要過早優化或實作當前不需要的功能。專注於當前的需求。

## 開發實務

1. **程式碼品質與風格**：
   - 遵循語言特定的風格指南（如 Python 的 PEP 8, JavaScript 的 ESLint 設定）。
   - 撰寫有意義的變數和函數名稱，讓程式碼本身具備文檔性（Self-documenting code）。
   - 進行適當的錯誤處理，避免靜默失敗。

2. **版本控制 (Git)**：
   - 頻繁提交（Commit often），確保每次提交都是原子性的（Atomic commits）。
   - 撰寫清晰的提交訊息（Commit message），說明「為什麼」修改，而不僅是「改了什麼」。
   - 善用分支（Branching）策略，如 Git Flow 或 GitHub Flow，來管理功能開發與發布。

3. **測試驅動開發**：
   - **單元測試 (Unit Testing)**：針對最小可測試單元進行驗證，確保邏輯正確。
   - **整合測試 (Integration Testing)**：驗證模組之間的互動。
   - **端對端測試 (E2E Testing)**：模擬使用者行為，驗證整個系統的運作。

## 協作與流程

1. **代碼審查 (Code Review)**：所有的程式碼變更在合併前都應經過他人審查。這不僅是為了發現錯誤，也是知識共享和保持代碼一致性的重要過程。
2. **持續整合與持續部署 (CI/CD)**：自動化構建、測試和部署流程。確保每次提交都能通過自動化測試，並能快速、可靠地部署到生產環境。
3. **技術債管理**：有意識地權衡快速交付與長期維護成本。定期安排時間重構代碼，償還技術債。
